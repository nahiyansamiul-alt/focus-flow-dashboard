# FocusFlow Windows Releases and Auto-Updates

FocusFlow publishes Windows releases from the public GitHub repository
[`CWE-119/focus-flow-dashboard`](https://github.com/CWE-119/focus-flow-dashboard).
The release workflow builds on Windows so the packaged SQLite module has the
correct native binary.

## Installer publishing and auto-updates are different

`npm run build:electron:win` only creates local artifacts in `dist_electron/`.
It does not publish a GitHub Release.

The Windows build contains two executables:

- `FocusFlow Setup <version>.exe` is the NSIS installer. Install this version to
  receive future in-app updates.
- `FocusFlow <version>.exe` is portable. It is convenient for one-off use, but
  it is not an auto-update target.

Auto-update also needs `latest.yml` and the NSIS `.blockmap` generated by the
same build. Do not manually mix metadata and installers from different builds.

## Automated release flow

The workflow at `.github/workflows/release-windows.yml` runs for tags matching
`v*.*.*`. It:

1. Verifies the tag equals `v` plus the version in `package.json`.
2. Installs the locked dependencies with `npm ci`.
3. Runs `npm run release:check`.
4. Builds the Windows targets and uploads them to a draft GitHub Release.
5. Publishes the release only after every artifact finishes uploading.

GitHub supplies the workflow's short-lived token. Never commit a GitHub token
or bundle one inside the application.

## Create a release

First merge and verify all intended changes on the canonical repository's
`main` branch. From a clean `main` checkout, create the next version commit and
tag. For the first corrected release, `minor` changes `1.1.0` to `1.2.0`:

```bash
npm run release:check
npm version minor -m "Release v%s"
git push origin main --follow-tags
```

Use `npm version patch` for a later bug-fix release such as `1.2.1`. Never reuse
a published version or tag: installed apps compare versions using semantic
versioning, and existing metadata contains checksums for a specific build.

Follow the tag workflow in GitHub Actions. When it completes, confirm the
release is published rather than draft and contains at least:

- the safe-named NSIS Setup executable referenced by `latest.yml`;
- its `.blockmap`;
- `latest.yml`;
- the portable executable, if you want to offer it as a separate download.

The local artifact can contain spaces while the GitHub asset uses a safe name
with hyphens. The electron-builder publisher handles that mapping. Uploading
only the local executable through the GitHub web UI can leave `latest.yml`
pointing to an asset that does not exist.

## Manual publishing fallback

Run this only on Windows and only when the tag workflow cannot be used. Create
a fine-grained GitHub token with Contents read/write access to this repository,
then keep it in the current PowerShell process:

```powershell
$env:GH_TOKEN = "your-temporary-token"
npm run release:check
npm run publish:electron:win
gh release edit "v1.2.0" --draft=false --latest
Remove-Item Env:GH_TOKEN
```

`publish:electron:win` explicitly passes `--publish always`. The release stays
draft during upload because `electron-builder.json` sets `releaseType` to
`draft`; the final `gh release edit` makes a complete release visible to apps.

## Test auto-update before announcing a release

A successful installer build is not enough. Test the real upgrade path on a
clean Windows VM:

1. Install the NSIS build for the previous version.
2. Publish a higher patch version from the same canonical repository.
3. Launch the installed older version.
4. Confirm the update-available and update-downloaded dialogs appear.
5. Choose **Install Now** and confirm the app restarts on the new version.
6. Confirm the backend starts and existing SQLite data remains intact.

Also inspect the packaged SQLite binding:

```bash
file dist_electron/win-unpacked/resources/app.asar.unpacked/node_modules/sqlite3/build/Release/node_sqlite3.node
```

It must report a Windows `PE32+` binary, not an `ELF` Linux binary.

## Existing 1.1.0 installations

The old 1.1.0 package was built with a placeholder update owner. Any user who
already installed that package must manually install the first corrected NSIS
release once. Releases after that can update normally through the app.

## Signing

The release configuration no longer disables Windows executable editing and
signing. Unsigned test releases can still be produced, but Windows SmartScreen
will warn users. Add the certificate and password as GitHub Actions secrets
named `WIN_CSC_LINK` and `WIN_CSC_KEY_PASSWORD` before treating the installer
as a polished public release; the Windows workflow already passes those secrets
to electron-builder when they are configured.

## Troubleshooting

- A draft release is invisible to installed apps. Publish it after all assets
  upload.
- The GitHub repository must remain public for token-free client updates.
- The package version must be higher than the installed version.
- `latest.yml`, the installer, and the blockmap must come from one build.
- Portable users must download a new portable executable manually.
- Run `npm run clean:electron` if stale local outputs are present. The cleanup
  script reclaims prior `dist_electron-trash-*` directories on a best-effort
  basis; close running packaged apps if Windows keeps files locked.
